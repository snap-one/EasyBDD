Easy BDD — framework comparison

How Easy BDD simplifies test authoring compared to traditional BDD (Behave/Cucumber) and Robot Framework.

No Python required TestRail-native Browser + API + Hardware
Three frameworks compared
Behave / Cucumber BDD

Gherkin feature files + Python step definitions. Two separate files per test. Engineers only.

Robot Framework

Keyword-driven .robot files with custom libraries. Verbose DSL, complex setup, heavy IDE dependency.

Easy BDD

Single YAML file per test. Written directly in TestRail. No coding, no setup, runs on Jenkins automatically.

Writing a browser login test
Behave — feature file
Feature: Login
  Scenario: Valid login
    Given I open the browser
    And I navigate to "${URL}"
    When I enter "admin" in "#user"
    And I enter "pass" in "#pass"
    And I click "#login-btn"
    Then I should see "Dashboard"
+ steps/login_steps.py (separate file)
@given('I open the browser')
def step(context):
    context.driver = webdriver.Chrome()

@when('I enter "{text}" in "{sel}"')
def step(context, text, sel):
    context.driver.find_element(
        By.CSS_SELECTOR, sel
    ).send_keys(text)
    
@then('I should see "{text}"')
def step(context, text):
    assert text in context.driver.page_source

Behave — feature file + 100+ line steps file
Feature: Firmware API
  Scenario: Get firmware info
    Given I have a valid auth token
    When I GET "/api/system/firmware"
    Then the status code should be 200
    And "partition1Version" should be
        present in "restful_res.fwConfs"
Robot Framework — RequestsLibrary
*** Settings ***
Library    RequestsLibrary

*** Test Cases ***
Get Firmware Info
    ${body}=    Create Dict
    ...    user=${USERNAME}
    ...    password=${PASSWORD}
    ${login}=    POST On Session
    ...    api    /system/login    json=${body}
    ${token}=    Set Variable
    ...    ${login.json()['restful_res']['token']}
    ${headers}=    Create Dict
    ...    Authorization=Bearer ${token}
    ${resp}=    GET On Session
    ...    api    /system/firmware
    ...    headers=${headers}
    Should Be Equal As Integers
    ...    ${resp.status_code}    200
Easy BDD — TestRail Preconditions only. No code files.
- api.request:
  method: POST
  url: ${api_url}/system/login
  body: {'user': '${api_username}',
    'password': '${api_password}'}
  store_as: login_response
- test.assert:
  expression: login_response.data
    .restful_res.errCode == 0
  message: Login should return errCode 0
- api.request:
  method: GET
  url: ${api_url}/system/firmware
  headers: {'Authorization': 'Bearer
    ${login_response.data
    .restful_res.token}'}
  store_as: firmware_response
- test.assert:
  expression: "'partition1Version' in
    firmware_response.data
    .restful_res.fwConfs"
  message: partition1Version should exist

No step library. No Python. No IDE.

Any QA engineer writes and modifies this directly in TestRail.

6 / 9
Variable management — sharing data across tests
Behave

Data flows via Python context object. Hooks written in environment.py. No cross-test shared variables without custom code.

Robot Framework

Variable files (.yaml or .py), suite-level variables, resource files. Complex scoping rules. IDE required to manage safely.

Easy BDD — Var: case pattern

One Var: case in the TestRail run holds all shared values. Every other case resolves ${variable_name} from it automatically. No code, no files.

base_url: http://192.168.100.3/cgi-bin/luci
username: araknis
password: SnapAV704
headless: False
retry: 0
no_datalake: True
Resolution priority (highest → lowest)

CLI overrides → store_as runtime values → test-level variables → Var: case → framework.yaml defaults

Assertion syntax — all three side by side
Behave (Python code)
assert resp.status_code == 200
data = resp.json()
rr = data['restful_res']
assert rr['errCode'] == 0
assert 'token' in rr
fwc = rr['fwConfs']
assert 'partition1Version' \
    in fwc

Should Be Equal
...  ${resp.status_code}
...  ${200}
Dictionary Should
...  Contain Key
...  ${resp.json()
...  ['restful_res']}
...  token
Easy BDD (TestRail)
- test.assert:
  expression:
    response.data
    .restful_res
    .errCode == 0

- test.assert:
  expression:
    "'token' in
    response.data
    .restful_res"
Dot-notation traversal — no brackets, no imports

firmware_response.data.restful_res.fwConfs.partition1Version == 'IMG-1.0.23.01'

Works directly in TestRail preconditions. No libraries, no step definitions, no Python knowledge required.

  • Self-healing selectors
  • Automatic HTML reports attached per run
  • Playwright recording → YAML conversion
  • Hardware and device API testing built-in
  • Run scheduling inside TestRail run description
